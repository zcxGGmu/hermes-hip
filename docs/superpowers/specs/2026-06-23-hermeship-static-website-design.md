# Hermeship 静态官网设计规格

## 目标

为 Hermeship 建一个独立的静态官网，第一眼呈现出清晰的开发者工具气质和操作面板感，结构上借鉴 `gajae-code.com` 的单页叙事节奏，但内容、命名和视觉资产完全使用 Hermeship 自身的表达。

这个站点的任务不是“讲故事”，而是让访客快速看懂三件事：Hermeship 是什么、它如何工作、它的真实边界在哪里。

## 非目标

- 不引入前端框架或构建工具。
- 不加入后端、表单、登录、埋点或动态数据。
- 不做多页营销站。
- 不复制外部站点的文案、品牌名或配色。
- 不把 README 内容原样搬进网页。

## 范围

实现范围限定在仓库内一个独立的静态站点目录，推荐为 `site/`。

页面只需要一个主入口 `site/index.html`，辅以少量 CSS 和极少量 JavaScript。页面内容以中文为主，保留少量英文命令名、文件名和技术名词；英文版 README 继续作为独立文档入口，而不是复制成整站第二套界面。

## 信息架构

站点采用单页、分章节滚动结构，章节顺序如下：

1. 顶部导航
2. Hero 首屏
3. 30 秒本地试跑
4. 工作流路径
5. 能力矩阵
6. 架构与图表
7. 运行边界与验证
8. Footer

导航只负责锚点跳转和语言/文档入口，不承担复杂菜单职责。移动端使用折叠式抽屉导航，桌面端使用横向导航。

## 视觉系统

页面整体是深色开发者工具风格，但不是暗黑营销页。

- 背景：接近黑色的冷灰或深蓝黑，保证高对比。
- 主强调色：Hermeship 现有品牌 lockup 的金黄/琥珀色，只用于品牌和少量高亮。
- 运行态颜色：绿色或青绿色，用于命令提示、成功状态和流程强调。
- 辅助强调色：冷蓝或青色，用于链接、焦点态和辅助标签。
- 文本：正文使用系统无衬线回退栈，命令、路径、事件名使用等宽字体栈。
- 形状：面板和卡片保持紧凑、克制，避免大面积圆角和装饰性浮层。
- 背景效果：只允许非常轻的渐变或描边，不使用漂浮光斑、泡泡或大块装饰性 orb。

视觉层级要像工具面板，不像宣传海报。页面应该密、清楚、可扫读。

## 组件结构

### 顶部导航

导航左侧放 Hermeship 标志或简化品牌块，右侧放章节锚点和 GitHub / README 入口。当前章节在滚动时要有明显状态反馈。

### Hero 首屏

首屏左侧是产品定位、简短说明和两个明确动作：

- 查看文档
- 本地试跑

首屏右侧使用一块高信息密度的“操作面板”视觉：

- 一段终端样式的命令区
- 一组状态标签
- 一张小型架构预览或图表缩略图

首屏必须在桌面和移动端都保留品牌识别，不允许标题、说明和视觉块互相挤压。

### 30 秒本地试跑

这一段是页面的实用入口，不是营销内容。它展示最少的本地启动命令，强调不需要外部凭据也能看到基本流程。

### 工作流路径

展示 Hermeship 的主流程：ingest -> normalize -> scrub -> route -> render -> deliver。

这一段应采用横向流程图或分段流程卡，移动端自动改为纵向堆叠。

### 能力矩阵

用表格或紧凑卡片列出：

- 默认具备的能力
- 需要显式启用的能力
- 仍在边界之外的能力

必须把真实边界说清楚，例如 observer plugin 的显式启用要求、Slack sink 不在默认范围、真实 live verification 不能被暗示为已完成。

### 架构与图表

使用仓库现有图表资产，优先展示：

- `hermeship-architecture.png`
- `hermeship-event-flow.png`
- `hermeship-observer-framework.png`
- `hermeship-github-discord-codex-workflow.png`

图表以 `figure` 方式呈现，每张图配简短说明，不做大段解释。桌面端可并排，移动端单列。

### 运行边界与验证

这一段明确告诉访客哪些事情 Hermeship 做，哪些事情不做，哪些结果只是本地 deterministic 验证，不是线上 live pass。

### Footer

Footer 只保留少量链接：README、English README、架构文档、GitHub 仓库。不要放多余宣传语。

## 交互与可访问性

- 所有导航和按钮都必须可键盘操作。
- 所有交互态都要有可见 focus。
- 所有图像都要有真实、具体的 alt 文本。
- 移动端不得出现横向滚动。
- 动效必须克制，并尊重 `prefers-reduced-motion`。
- 触控目标必须足够大，不能依赖 hover 才能理解操作。
- 代码块和命令块必须可选中、可复制，复制动作应是可选增强，不是页面核心依赖。

## 资产契约

站点优先复用现有仓库资产，不重新生成视觉素材。

- `docs/assets/branding/hermeship-lockup.png` 用作品牌主图或首屏标识。
- `docs/assets/branding/hermeship-icon.png` 用作 favicon 或移动端品牌块。
- `docs/assets/diagrams/*.png` 用作架构、流程和边界展示。

如果某个图表在页面上不够清晰，优先调整版式和尺寸，不优先重绘。

## 安全与隐私

页面保持静态，不引入第三方脚本、统计埋点或远程运行时。链接和按钮都只做明确可见的导航动作，不在后台发起隐式请求。

## 验证策略

上线前至少确认以下几点：

- 桌面和移动端布局没有重叠、截断或横向滚动。
- 导航锚点可以正常跳转。
- 图像和代码块可以正确显示。
- 对比度和 focus 状态足够清楚。
- `prefers-reduced-motion` 下页面仍然可读。

## 发布边界

该站点是 Hermeship 的独立静态入口，不是对外 API，不承担运行时行为，也不改变 Rust daemon 的任何逻辑。它只是把 Hermeship 的定位、流程、资产和边界用更适合浏览的方式呈现出来。
